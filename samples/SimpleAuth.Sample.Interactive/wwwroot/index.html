🔍 Discovery & JWKS

OpenID Connect Discovery 1.0 — server metadata and public keys.

Discovery Document

GET /.well-known/openid-configuration — Advertises all endpoints, supported features, algorithms.

JSON Web Key Set (JWKS)

GET /.well-known/jwks.json — Public keys used to verify token signatures.

🔐 Authorization Code + PKCE

Full OAuth 2.1 authorization code flow with PKCE (S256). Includes login, consent, and token exchange.

Flow Configuration

Steps

Generate PKCE

Create code_verifier (random 43 chars) and code_challenge (SHA-256 of verifier, base64url).

Start Authorization

Redirect to /connect/authorize with parameters. Opens login → consent → callback.

Exchange Code for Tokens

POST /connect/token with the authorization code + code_verifier.

Token Response

Decoded ID token and access token.

🤖 Client Credentials

Machine-to-machine token issuance. No user involved — the client authenticates directly.

🔄 Refresh Token

Exchange a refresh token for new access + refresh tokens. Demonstrates token rotation.

Refresh Token Exchange

First complete the Authorization Code flow to obtain a refresh token.

POST /connect/introspect

POST /connect/revocation

🛡️ DPoP (Proof-of-Possession)

RFC 9449 — Bind tokens to a client's cryptographic key. Prevents token theft/replay.

DPoP Flow

Generate EC P-256 Key Pair

Uses Web Crypto API to create an ephemeral signing key in the browser.

Create DPoP Proof JWT

Sign a proof JWT with htm, htu, iat, jti claims.

Request Token with DPoP

POST /connect/token with DPoP header. Token will be bound to the key (cnf.jkt).

📨 PAR (Pushed Authorization Request)

RFC 9126 — Push authorization parameters to the server first, then redirect with a request_uri.

PAR Flow

Push Authorization Request

POST /connect/par with all authorize parameters. Server returns a request_uri.

Redirect with request_uri

Open /connect/authorize?client_id=...&request_uri=... (parameters are loaded server-side).

Exchange Code

Same token exchange as normal auth code flow.

📄 Form Post Response Mode

OAuth 2.0 Form Post Response Mode — authorization response delivered via HTML form POST instead of query redirect.

Start Authorization with response_mode=form_post

The server returns an auto-submitting HTML form that POSTs the code to the redirect_uri.

🚪 End Session (Logout)

OIDC RP-Initiated Logout — terminates the user session and optionally redirects.

GET /connect/endsession

⚡ prompt=none (Silent Auth)

OIDC §3.1.2.1 — Check if user is already authenticated without showing UI. Returns login_required if not.

Silent Authentication Check

Adds prompt=none to the authorize request. If user has an active session, returns a code silently. Otherwise returns login_required error.
